## 2. WBS 分解思路
本项目WBS采用面向交付物的方法进行分解，确保每一项均为可检查、可交付的成果，而非过程性活动。  

分解逻辑如下：  
 一级：项目整体成果（受控配置项）  
二级：按成果类型划分（仓库、WBS文档、进度计划、配置管理、总结报告）  
三级：细化为最小可管理成果单元（如清单、说明、记录等）  

例如：  
WBS文档 → 成果分解表 → L3成果清单  
配置管理 → 冲突处理 → 冲突解决记录  

该分解方式保证了：  
结构完整  
逻辑清晰  
可直接支撑进度计划编制  

## 3. 进度安排依据
进度计划基于WBS结果提取关键活动，并结合任务依赖关系进行安排。

主要依据包括：
1.任务依赖关系
WBS完成后才能制定进度计划
进度计划完成后才能进入总结阶段
2.关键路径识别
关键活动包括：
需求确认
WBS编制
进度计划制定
冲突实验
总结报告
3.工期设定
所有活动均按实验时间（8学时）进行简化安排，保证每项任务均可在合理时间内完成
4.责任分配
每个活动明确负责人，提高执行效率

## 4. 配置管理与协作过程
本项目采用Git进行配置管理，建立了规范的分支与协作机制：

（1）配置项识别

主要配置项包括：
README.md
WBS文档
进度计划文档
配置管理文档
冲突文件（conflict.md）
总结报告
（2）分支策略
main：最终成果分支
dev：阶段集成分支
feature-*：成员开发分支
（3）协作流程
成员在各自 feature 分支进行修改
提交代码并发起 Pull Request
配置管理员进行审核与合并
必要时进行分支同步（main → feature）
（4）提交规范
提交信息清晰（如 Update / Merge）
提交人统一使用拼音
所有成果最终合并至 main 分支
